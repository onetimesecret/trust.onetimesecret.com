# Cryptographic rationale for the compelled-disclosure and passphrase language

Extracted 2026-07-02 from `onetime-privacy-statement-review-tasks.md` (TODO-8,
BLK-2, BLK-5) before that working file was deleted; its full text remains in
git history (last committed state: commit 8d02ba6). This is the sole recorded
rationale for the passphrase and compelled-disclosure wording in the Privacy
Statement ("How we respond to compelled disclosure") and DPA Schedule C §1.1.

## Why the "we cannot decrypt passphrase-protected secrets" claim was removed

Codebase audit (verified against Familia v2.3.3):

- **v1 (legacy):** the passphrase IS part of key derivation
  (`encryption_key_v2` includes `passphrase_temp`). Without the passphrase,
  decryption is impossible. The old claim was accurate for v1 only.
- **v2 (Familia `encrypted_field`):** `encrypted_field :ciphertext` at
  `secret.rb:43` has no `aad_fields`. The declared
  `transient_field :ciphertext_passphrase` is never read. Encryption at
  `receipt.rb:218` sets `secret.ciphertext = content` without the
  passphrase. The passphrase is Argon2-hashed for access control only
  (`update_passphrase`). For v2, OTS CAN decrypt passphrase-protected
  secrets using only the system key.

## Why AAD binding cannot restore the "cannot decrypt" property

Binding the passphrase via `aad_fields` is cryptographically insufficient.
AAD in XChaCha20-Poly1305 is authenticated but not used for key derivation.
Confidentiality depends on key + nonce; the keystream is
`XChaCha20(key, nonce)`. A key-holder can recover plaintext by computing the
keystream and XORing, ignoring the auth tag entirely. AAD binding only makes
the normal verifying API refuse — it does not remove the capability to
decrypt. The compelled-disclosure threat model assumes the key-holder is
under court order; "cannot" is a capability statement, and AAD does not
remove capability.

## Resolution (Option B, doc fix — decided)

Align the Privacy Statement to the DPA, which already told the truth
(Schedule C §1.1: "The passphrase is not incorporated into the encryption
key derivation; it controls authorization to decrypt, not the cryptographic
ability to do so"). Remove the "cannot decrypt" claim; state instead that a
passphrase is an access-control gate verified before decryption, and that
decryption keys are not available on the database machine where the data is
stored. The alternative (Option A: put the passphrase into key derivation
via Familia `build_context`) was rejected — bigger change, needs a version
gate for in-flight secrets, and a forgotten passphrase would mean
permanently unrecoverable data.

## Proxy clause — removed as describing a nonexistent flow

The clause "encrypted client-side with a key we never receive, such as in
flows where we act as a proxy for re-encrypted ciphertext" described a flow
that does not exist in the codebase. Two code paths write
`secret.ciphertext` — both receive plaintext. No API endpoint accepts
pre-encrypted content.

## Related facts

- FEDERATION_SECRET is for cross-region email hashing (ADR-008), not
  secret-identifier verification; the "confirm whether the secret identifier
  was generated by us" capability may not exist, so that clause was removed
  from the compelled-disclosure section.
- Catalyst Cloud datacenter verification: Catalyst Cloud operates
  datacenters in Wellington, Porirua, and Hamilton per their public
  infrastructure page ("Porirua" is the correct spelling used in the
  region tables).
- Credential hashing (from onetime-dpa-review-tasks.md XD-1): Argon2id is
  the default for new credentials; legacy BCrypt (cost 12) hashes were
  retained for verification at audit time. See `../REMAINING_DECISIONS.md`
